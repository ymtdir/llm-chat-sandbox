# コミット規約

## プレフィックス

| プレフィックス | 説明                                                       | 使用例                                      |
| -------------- | ---------------------------------------------------------- | ------------------------------------------- |
| `feat:`        | 新機能追加                                                 | feat: ユーザー認証機能を実装                |
| `fix:`         | バグ修正                                                   | fix: ログイン時のセッション保存エラーを修正 |
| `docs:`        | ドキュメント                                               | docs: READMEにインストール手順を追加        |
| `style:`       | UI/UX の変更（見た目・デザインのみ）                       | style: ヘッダーのレイアウトを改善           |
| `refactor:`    | リファクタリング・パフォーマンス改善・テスト/CI 環境整備   | refactor: データベースクエリを最適化        |
| `test:`        | テストコード追加・修正                                     | test: ログイン機能のE2Eテストを追加         |
| `chore:`       | 依存更新・フォーマッタ整形等の雑務                         | chore: 依存パッケージを更新                 |

**ラベル対応**: `feat:` → `enhancement`、`fix:` → `bug`、`style:` → `ui/ux`、`docs:` → `documentation`、`refactor:` → `refactor`。`test:` と `chore:` はラベル自動付与対象外（詳細は [label-definitions.md](./label-definitions.md) 参照）。

## コミットメッセージ規則

```
[prefix]: [日本語での説明]
```

## 作業単位の定義

**良いコミット単位**:

- 1つの関数の追加
- 1つのコンポーネントの修正
- 1つのテストケースの追加
- 定数やタイプ定義の追加
- インポート文の整理

**避けるべき**:

- 複数機能をまとめた大きなコミット
- 「WIP」や「temporary」のようなコミット
- 動作しない中途半端な状態でのコミット（ただし後述の TDD 例外あり）

**TDD の赤コミット例外**: テスト先行開発で「失敗するテスト」を先にコミットする場合は許容する。メッセージに状態を明示する: 例 `test: validateEmail の空文字ケースを追加 (RED)`。次のコミットで `fix:` / `feat:` により緑化すること。

## 作業単位の例

### バグ修正の場合

```bash
fix: エラーメッセージの定数を定義
fix: バリデーション関数を修正
test: エッジケースのテストを追加
fix: コンポーネントに修正を適用
docs: 変更内容をコメントで説明
```

### 新機能追加の場合

```bash
feat: 新しいAPIエンドポイントを定義
feat: データモデルを作成
feat: サービス層のロジックを実装
test: サービスのユニットテストを追加
feat: UIコンポーネントを作成
test: コンポーネントのテストを追加
docs: API仕様書を更新
```

### リファクタリングの場合

```bash
refactor: 共通処理を抽出
refactor: 型定義を整理
refactor: 不要なコードを削除
test: リファクタリング後のテストを確認
chore: インポート文を整理
```

## 注意事項

- **1コミット1目的**: 各コミットは1つの明確な目的を持つ
- **日本語で記載**: チームメンバーが理解しやすいように
- **具体的に**: 「修正」ではなく「何を修正したか」を書く
- **作業単位**: 機能単位ではなく、実際の作業単位でコミット
