# PR Review スキル

PRのラベルに応じて専門スペシャリストの観点を選択し、領域に特化したレビューを行うスキルです。

## スペシャリスト一覧

| スペシャリスト | ファイル                        | 概要                                         |
| -------------- | ------------------------------- | -------------------------------------------- |
| アーキテクチャ | `./specialists/architecture.md` | 設計パターン・モジュール構造・依存関係を評価 |
| セキュリティ   | `./specialists/security.md`     | 脆弱性・認証認可・機密情報露出を検証         |
| UI/UX          | `./specialists/ui-ux.md`        | 一貫性・アクセシビリティ・ユーザー体験を評価 |
| コード品質     | `./specialists/code-quality.md` | 可読性・命名・DRY・複雑度を評価              |
| パフォーマンス | `./specialists/performance.md`  | 処理効率・リソース使用量・応答速度を評価     |
| テスト         | `./specialists/testing.md`      | テストの網羅性・品質・信頼性を評価           |

## ラベルとスペシャリストの対応

PRのラベルに応じて、重点的にレビューするスペシャリストを選択します。
ラベルの詳細は `../../rules/label-definitions.md` を参照してください。

| ラベル        | 重点スペシャリスト             | 補助スペシャリスト |
| ------------- | ------------------------------ | ------------------ |
| bug           | セキュリティ, テスト           | コード品質         |
| enhancement   | アーキテクチャ, テスト         | コード品質         |
| ui/ux         | UI/UX, コード品質              | テスト             |
| documentation | ―                              | ―                  |
| refactor      | アーキテクチャ, パフォーマンス | コード品質, テスト |

- **重点スペシャリスト**: ガイドの全チェック項目を網羅的にレビューし、必須・推奨・提案すべてのレベルで指摘する
- **補助スペシャリスト**: ガイドのチェック項目のうち「必須」レベルの問題のみを指摘する。推奨・提案レベルは省略
- `documentation` ラベルのPRはコードレビュー不要。内容の正確性のみ確認

## レビュープロセス

### ステップ1: PR情報の取得

```bash
gh pr view [番号] --json number,title,body,labels,headRefOid,files
gh pr diff [番号]

# インラインコメント投稿時に必要な owner / repo を取得
gh repo view --json owner,name --jq '"\(.owner.login)/\(.name)"'
```

`headRefOid` はインラインコメント投稿時の `commit_id` として使用する。

### ステップ2: ラベルの確認とフロー選択

1. PR のラベルを取得する
2. **`documentation` ラベルの場合**: ステップ4-6 を実行せず、後述「documentation レビュー」節に従う
3. **それ以外のラベル / ラベル未設定の場合**: 上記対応表に従いスペシャリストを選択してステップ3 以降に進む
   - ラベル未設定なら全 6 スペシャリストを補助レベルで実施

### ステップ3: 仕様との整合確認（共通）

すべてのレビューで、変更内容が以下と矛盾しないかを確認する：

- 関連Issueの要件
- `docs/requirements.md`（存在する場合）
- `docs/basic-design.md`（存在する場合）
- `docs/detailed-design.md`（存在する場合）

仕様との不一致・要件の欠落・スコープ外の変更があれば、最初のスペシャリストの総評に含める。
上記ドキュメントが存在しない場合はスキップする。

### ステップ4〜6: スペシャリストごとのレビューと投稿

**選択された各スペシャリストごとに、以下を繰り返す。**
各スペシャリストが独立したレビュワーとしてGitHubに投稿するため、スペシャリストの数だけレビューが付く。

#### 4. 変更ファイルの分析

当該スペシャリストの観点で変更内容を分析し、問題を検出する。

- **指摘あり**: ステップ5（インライン）・6（総評）を両方実行
- **指摘なし**: ステップ5 はスキップ、**ステップ6 の総評は必ず投稿**（評価 `✅ 問題なし`、「改善提案」節は省略し「良い点」のみ記載）

補助スペシャリストは「必須レベルの問題のみ」が指摘対象。推奨・提案レベルは総評の「改善提案」から省略するが、「良い点」は重点/補助を問わず記載する。

#### 5. インラインコメントの投稿

問題のある箇所には行単位でコメントを投稿する。
他のスペシャリストが同じ行に既にコメント済みの場合は重複を避ける。

```bash
gh api repos/[owner]/[repo]/pulls/[番号]/comments \
  --method POST \
  -f body="[改善提案]" \
  -f commit_id="[headRefOid]" \
  -f path="[ファイルパス]" \
  -f line=[行番号] \
  -f side="RIGHT"
```

#### 6. 総評コメントの投稿

スペシャリストごとに個別の総評を投稿する：

```bash
gh pr review [番号] --comment --body "[スペシャリスト別レビュー結果]"
```

## documentation レビュー

`documentation` ラベルの PR はコードレビュー不要。以下の観点で内容の正確性のみ確認し、総評 1 本を投稿する（インラインコメントは投稿しない）。

### 確認観点

- **事実整合**: バージョン番号・コマンド・API 仕様等がリポジトリの実体（`package.json`, 設定ファイル, CI 定義等）と一致しているか
- **参照整合**: 参照されるファイル（`.nvmrc` など）や外部リンクが存在・有効か
- **削除妥当性**: 削除された記述が本当に不要か（他箇所から参照されていないか）
- **記述一貫性**: 他ドキュメント・CI 設定・実コードとの記述が矛盾しないか

### 投稿フォーマット

```bash
gh pr review [番号] --comment --body "$(cat <<'EOF'
## ドキュメント内容の正確性確認

**評価**: ✅ 問題なし / ⚠️ 改善推奨 / ❌ 要修正

### 確認できた点

- [正確性が確認できた事項]

### 要確認事項

- [コミッター側で実体と突き合わせて確認してほしい項目]

### 提案

- [任意の改善提案]
EOF
)"
```

## 出力フォーマット

スペシャリストごとに以下のフォーマットで投稿する。

    ## [スペシャリスト名] レビュー

    **評価**: ✅ 問題なし / ⚠️ 改善推奨 / ❌ 要修正

    ### 良い点

    - [具体的な良い実装]

    ### 改善提案

    #### 必須

    - [重要な修正点]

    #### 推奨

    - [改善したほうが良い点]

    #### 提案

    - [検討してみてください]

## レビュー基準

各スペシャリストが独立して評価を付ける。

| 評価        | 条件                                           |
| ----------- | ---------------------------------------------- |
| ✅ 問題なし | 当該領域で重大な問題なし                       |
| ⚠️ 改善推奨 | 軽微な改善点あり（マージは可能）               |
| ❌ 要修正   | 当該領域で重大な問題あり（マージ前に修正必要） |

## トラブルシューティング

| 症状                   | 対処                                                    |
| ---------------------- | ------------------------------------------------------- |
| PR情報取得エラー       | `gh auth status`で認証確認                              |
| インラインコメント失敗 | `commit_id`が正しいか確認。PRに新しいpushがあると変わる |
| ラベルが未設定         | 全スペシャリストを補助レベルで実施                      |
| diffが大きすぎる       | 重要度順にファイルを優先レビュー                        |
