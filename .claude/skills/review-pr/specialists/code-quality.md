# コード品質

コードの可読性・保守性・一貫性を評価します。

## クリーンコード原則

### 命名

- 名前は意図を明示する。コメントが必要な名前は失格
- 汎用名（`x`, `data`, `list`）や型プレフィックス（`accountList`）を避ける
- 関数は動詞（`getUser()`, `validateEmail()`）、クラスは名詞を使う
- 発音しやすく、検索しやすい名前にする

### 関数

- 小さく保つ。引数は3つ以下、20行以内が理想
- 1つのことだけをうまくやる（単一責任）
- ネストは2〜3段まで。深くなったら関数を抽出する

### コメント

- コード自体が語るべき。コメントは「なぜ」を説明する場合のみ
- 意図の明確化・非自明な副作用の警告にのみ使用する
- 「何をしているか」のコメントはコード自体を改善すべきサイン

## 複雑度の判断基準

| 種類             | 測定対象                 | レビューでの使い方                                 |
| ---------------- | ------------------------ | -------------------------------------------------- |
| **循環的複雑度** | 分岐の数（テスト容易性） | テストケース数の目安。10超は要注意                 |
| **認知的複雑度** | 理解に要する精神的負荷   | 可読性の指標。ネストの深さ・制御フローの中断が主因 |

認知的複雑度を優先して確認する。認知的複雑度が高いコードは変更時にバグを生みやすい。

### 複雑度が高い兆候

- ネストが3段以上（各段で認知負荷が倍増）
- `break`, `continue`, 早期リターンが乱立
- 複合条件式（`&&`, `||`の連鎖）
- 長い引数リストと条件分岐チェーン

## コードスメル

以下のスメルを検出した場合、対応するリファクタリングを提案する。

| スメル                 | 兆候                             | リファクタリング                                            |
| ---------------------- | -------------------------------- | ----------------------------------------------------------- |
| **長いメソッド**       | 20行超の関数                     | **メソッド抽出**: 1つのタスクを行うブロックを名前付き関数へ |
| **巨大クラス**         | 200行超・責務が複数              | **クラス分割**: 責務ごとに分離                              |
| **プリミティブ執着**   | 文字列で電話番号・メールを扱う   | **クラスで置換**: ドメイン固有の型を導入                    |
| **データの群れ**       | 同じ変数群が常にセットで渡される | **パラメータオブジェクト導入**: 値オブジェクトにまとめる    |
| **マジックナンバー**   | ハードコードされた数値・文字列   | **定数化**: 名前付き定数に置き換える                        |
| **重複コード**         | コピー&ペーストのパターン        | **共通ロジック抽出**: 関数やモジュールに集約                |
| **死んだコード**       | 使われていないコード・変数       | **削除**: 残しておく理由がなければ除去                      |
| **巨大コンポーネント** | 200行超のコンポーネント          | **コンポーネント分割**: UI・ロジック・副作用を分離          |
| **Props過多**          | propsが5つ以上                   | **コンポーネント分割**または**オブジェクトにまとめる**      |

## チェック項目

- [ ] 命名規則がプロジェクトの慣習と一貫しているか
- [ ] 関数・メソッドの責務が単一で明確か
- [ ] DRY原則が守られているか（不要な重複がないか）
- [ ] 認知的複雑度が適切な範囲か（ネスト3段以上は要注意）
- [ ] 上記のコードスメルに該当する箇所がないか

## レビュー手順

1. 変更ファイルのdiffを確認する
2. 既存コードの命名・構造の慣習を把握する
3. クリーンコード原則・コードスメルに照らして問題を検出する
4. 問題を検出した場合は具体的なリファクタリングを提案する

## 指摘の観点

- **命名の不整合**: プロジェクトの命名規則と異なる変数名・関数名
- **責務の肥大化**: 1つの関数・クラスに複数の責務が詰め込まれている
- **コードの重複**: 同一ロジックが複数箇所に存在する
- **過剰な複雑度**: ネストが深い・条件分岐が多すぎる処理
