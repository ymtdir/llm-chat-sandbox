# アーキテクチャ

設計パターン・モジュール構造・依存関係の適切性を評価します。

## 基本原則: SOLID

レビュー時に以下の原則への違反がないかを確認する。

| 原則                           | 確認ポイント                                                                           |
| ------------------------------ | -------------------------------------------------------------------------------------- |
| **単一責任 (SRP)**             | クラスやモジュールに変更理由が複数ないか。`Manager`や`Handler`等の汎用名は違反の兆候   |
| **開放閉鎖 (OCP)**             | 新機能追加時に既存コードのif/else分岐を増やしていないか。拡張で対応すべき              |
| **リスコフ置換 (LSP)**         | サブクラスが親クラスの契約を破っていないか。「未実装」例外を投げるオーバーライドは違反 |
| **インターフェース分離 (ISP)** | 使わないメソッドの実装を強制していないか。巨大なインターフェースは分割すべき           |
| **依存性逆転 (DIP)**           | 上位モジュールが下位モジュールの具体実装に直接依存していないか。抽象に依存すべき       |

## アーキテクチャのアンチパターン

以下のパターンを検出した場合は指摘する。

- **God Class**: 400行超・10以上のpublicメソッド・複数の関心事が混在。責務を分割すべき
- **Feature Envy**: 他オブジェクトのデータを過度に参照するメソッド。データの近くにロジックを移動すべき
- **Shotgun Surgery**: 1つの変更が多数のファイルに波及する。関連機能の凝集度を高めるべき
- **循環依存**: A→B→Aの相互依存。共有インターフェースの抽出やMediator導入で解消すべき
- **Leaky Abstraction**: 下位層の実装詳細が上位層に露出。リポジトリがSQLを隠蔽できているか確認

## 依存関係のルール

- **依存の方向**: UI層→サービス層→リポジトリ層→データ層。逆方向の依存は禁止
- **非循環依存の原則 (ADP)**: モジュールの依存グラフはDAG（有向非巡回グラフ）でなければならない
- **依存性注入**: コンストラクタで依存を受け取る。内部でインスタンスを生成しない

## コンポーネント指向設計

React・Vue・Svelte等のコンポーネントベースのアーキテクチャに適用する原則。

### コンポーネント分割

- **単一責任**: 1コンポーネントが1つの役割を持つ。表示・ロジック・データ取得が混在していないか
- **適切な粒度**: 巨大コンポーネント（200行超）は分割すべき。逆に過剰な分割は追跡困難になる
- **Presentational / Container分離**: UIの見た目（Presentational）とデータ・ロジック（Container/Hook）を分離しているか

### 状態管理

| スコープ               | 適切な手法                        | アンチパターン                          |
| ---------------------- | --------------------------------- | --------------------------------------- |
| コンポーネントローカル | useState / ref                    | ローカルで済む状態をグローバルに置く    |
| 親子間                 | Props / イベント                  | 2階層超のProps drilling                 |
| 横断的                 | Context / Store（Redux, Pinia等） | あらゆる状態をグローバルStoreに集約     |
| サーバー状態           | TanStack Query / SWR等            | サーバーデータをローカルstateで手動管理 |

### ロジックの分離

- **カスタムフック / コンポーザブル**: UIから独立したロジック（データ取得、バリデーション、ステート操作）はカスタムフックに抽出されているか
- **副作用の管理**: useEffect / onMounted等の副作用がコンポーネントに散在していないか。依存配列は正しいか

### コロケーション

- 関連するファイル（コンポーネント、テスト、スタイル、型定義）が近くに配置されているか
- 機能単位でディレクトリを切る（Feature-based）か、種類で切る（Type-based）か、プロジェクトの方針と一致しているか

## 凝集度と結合度

- **高凝集**: モジュール内の要素が密接に関連し、1〜2文で責務を説明できるか
- **低結合**: モジュール間の変更が波及しないか。importの数やfan-outが過剰でないか

## チェック項目

- [ ] SOLID原則に違反する設計がないか
- [ ] 上記のアンチパターンに該当する箇所がないか
- [ ] モジュール間の依存方向が正しいか（循環依存がないか）
- [ ] 責務の分離が正しく行われているか（レイヤー違反がないか）
- [ ] コンポーネントの粒度・分割が適切か（フロントエンドの場合）
- [ ] 状態管理のスコープが適切か（ローカル/グローバルの使い分け）
- [ ] 既存のインターフェース・契約を破壊していないか

## レビュー手順

1. 変更ファイルのディレクトリ構造とモジュール配置を確認する
2. import/依存関係を分析し、レイヤー間の方向を検証する
3. SOLID原則への違反とアンチパターンの有無を確認する
4. 破壊的変更の有無を評価する

## 指摘の観点

- **レイヤー違反**: プレゼンテーション層からDB直接アクセス等の越境
- **循環依存**: モジュール間の相互依存が発生している
- **責務の誤配置**: ファイルやモジュールの責務と一致しないロジックの配置
- **破壊的変更**: 既存の公開インターフェースを予告なく変更している
- **SOLID違反**: 上記原則に反する設計判断
- **コンポーネント肥大化**: 巨大コンポーネントにUI・ロジック・データ取得が混在
- **状態スコープの誤り**: ローカルで済む状態のグローバル化、または逆のケース
