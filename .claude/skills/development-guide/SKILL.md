# 開発ガイドライン作成スキル

高品質な開発ガイドラインを作成するためのスキルです。要件定義書・基本設計書・詳細設計書の内容を元に、具体的なリポジトリ構造・開発ルール・プロジェクト用語を定義します。

## 前提条件

- **必須**: `docs/requirements.md`（要件定義書）が作成されている
- **必須**: `docs/basic-design.md`（基本設計書）が作成されている
- **必須**: `docs/detailed-design.md`（詳細設計書）が作成されている

## 出力ルール

`../../rules/document-generation-guide.md` を参照。

**対象ファイル**: `docs/development-guide.md`

## 参照ファイル

- **テンプレート**: `./assets/template.md`
- **リポジトリ構造ガイド**: `./references/structure-guide.md`
- **実装ガイド**: `./references/implementation-guide.md`
- **プロセスガイド**: `./references/process-guide.md`
- **用語集ガイド**: `./references/glossary-guide.md`

## 開発ガイドライン作成プロセス

### 1. 入力の確認

- `docs/requirements.md` / `docs/basic-design.md` / `docs/detailed-design.md` を読み、技術スタック・アーキテクチャ・主要モジュールを把握する
- 既存の `docs/development-guide.md` があるか確認。ある場合は `../../rules/document-generation-guide.md` の「既存ドキュメント最優先」ルールに従い、既存の節構造・表記・言語を維持したまま更新する
- 前提ドキュメントが存在しない場合は下記トラブルシューティングに従う（skill を中止）

### 2. 開発ガイドラインドラフトの生成

3 つの設計ドキュメントをもとに、テンプレート（`./assets/template.md`）と参照ガイド（structure/implementation/process/glossary）に従ってガイドラインを生成します。

行数の目安は **300-400 行以内（上限目安、簡潔優先）**。超過する場合は `../../rules/document-generation-guide.md` に従いサブディレクトリ（`docs/guide/`）に分割する。

### 3. 開発ガイドラインのレビューと改善

生成されたガイドラインを以下 5 観点でレビューし、問題があれば修正する:

1. リポジトリ構造の妥当性（詳細設計のモジュール構成と整合しているか）
2. コーディング規約の網羅性（言語別ルール・命名・フォーマッタ・リンタ設定）
3. 開発プロセス（ブランチ運用・コミット規約・PR/レビュー・テスト・デプロイ）
4. 用語集の完全性（要件・設計に出てくる主要用語を網羅）
5. 要件・設計とのトレーサビリティ（3 ドキュメントの主要決定がガイドラインに反映されているか）

**終了条件**: 品質チェックリスト（本 skill 末尾）の全項目が ○ になり、かつユーザー（人間）が承認した時点で終了する。

**レビュー結果の扱い**: レビュー結果・改善ログは `docs/development-guide.md` 本体には含めず、会話上のログとしてユーザーに提示する。

**注意点**: AI のレビューを鵜呑みにせず、最終的な判断は必ず人間が行う

## 既存ドキュメント更新時の整合判断

既存の `docs/development-guide.md` があり、新しい設計と関係する場合:

- **設計に既存ガイドの廃止が明示されている**: 置換
- **廃止記述がない**: 併存（既存を残し、新ルールを追記して関係を注記）
- **スタイル整合**: 既存が英語節・箇条書き主体なら新規追記もそれに合わせる（template の日本語節構造を強制しない）

## 品質チェックリスト

### リポジトリ構造

- [ ] ディレクトリ構成図が提示されているか
- [ ] 各ディレクトリの責務が記載されているか
- [ ] 詳細設計のモジュール構成と整合しているか

### コーディング規約

- [ ] 使用言語ごとの規約（命名・フォーマッタ・リンタ）が記載されているか
- [ ] コメント・ドキュメンテーション方針が記載されているか
- [ ] 型・エラー処理・テストの書き方のガイドが含まれるか

### 開発プロセス

- [ ] ブランチ戦略が明記されているか
- [ ] コミット規約（`../../rules/commit-conventions.md` 参照）への言及があるか
- [ ] PR/レビューのフローが記載されているか
- [ ] ローカル開発・テスト・CI/CD 手順が含まれるか

### 用語集

- [ ] 要件・設計ドキュメントで使われる主要用語（ドメイン用語・技術用語）が定義されているか
- [ ] 略語の初出に正式名称が併記されているか

### トレーサビリティ

- [ ] 3 ドキュメントの主要決定（技術スタック・レイヤー構成・モジュール境界）がガイドラインに反映されているか

## トラブルシューティング

- **前提ドキュメントが不足**: 要件定義書・基本設計書・詳細設計書が必要です。不足している skill（`requirements-definition` / `basic-design` / `detailed-design`）を先に実行してから再実行してください。
- **既存のガイドとの競合**: `../../rules/document-generation-guide.md` の優先順位ルールに従ってください（既存ドキュメント最優先）。
