# 用語集作成ガイド

## 基本原則

### 1. 明確で一貫した定義

用語の定義は曖昧さを排除し、誰が読んでも同じ理解ができるようにします。

**悪い例**:

    ## タスク

    ユーザーがやるべきこと

**良い例**:

    ## タスク (Task)

    **定義**: ユーザーが完了すべき作業の単位。タイトル、説明、期限、
    ステータス(未着手/進行中/完了)を持つ。

    **関連用語**: サブタスク、タスクグループ

    **使用例**:

    - 「タスクを追加する」: 新しいタスクをシステムに登録する
    - 「タスクを完了する」: タスクのステータスを完了に変更する

### 2. 具体例を含める

抽象的な定義だけでなく、具体的な使用例を示します。

### 3. 関連用語をリンクする

用語同士の関係性を明確にします。

**例**:

    ## タスク (Task)
    **定義**: [定義]

    **関連用語**:
    - [サブタスク](#サブタスク): タスクを細分化したもの
    - [タスクグループ](#タスクグループ): 複数のタスクをまとめたもの

    **親子関係**:
    - 親: タスクグループ
    - 子: サブタスク

## 用語の分類方法

### ドメイン用語の定義

**対象**: プロジェクト固有のビジネス概念

**定義すべき項目**:

- 定義（1-2文での簡潔な定義）
- 説明（詳細な説明、背景、制約）
- 関連用語
- 使用例
- データモデル（該当するファイルパス）

### 技術用語の定義

**対象**: 使用している技術、フレームワーク、ツール

**定義すべき項目**:

- 定義（技術の簡潔な説明）
- 本プロジェクトでの用途
- バージョン
- 選定理由
- 関連ドキュメント

**例**:

    ## React

    **定義**: UIコンポーネントを構築するためのJavaScriptライブラリ

    **本プロジェクトでの用途**:
    フロントエンドのUI構築に使用。

    **バージョン**: 19.x

    **選定理由**:
    - コンポーネントベースの開発による再利用性
    - 豊富なエコシステム

    **関連ドキュメント**:
    - [基本設計書](./basic-design.md#技術スタック)
    - [開発ガイドライン](./development-guide.md#コーディング規約)

### 略語・頭字語の定義

**原則**:

- 正式名称を明記
- 初出時は略語と正式名称を併記
- プロジェクト固有の略語は避ける（一般的な略語のみ使用）

**例**:

    ## API

    **正式名称**: Application Programming Interface

    **意味**: ソフトウェア間の通信インターフェース

    **本プロジェクトでの使用**:
    RESTful APIとしてバックエンドサービスを提供。

## 状態遷移の定義

**対象**: エンティティのステータスや状態

**定義方法**:

1. **表形式で列挙**
2. **遷移条件を明記**
3. **Mermaid図で可視化**

**例**:

    ## 注文ステータス (Order Status)

    **定義**: 注文の処理状態を示す列挙型

    | ステータス  | 意味     | 遷移条件           | 次の状態           |
    | ----------- | -------- | ------------------ | ------------------ |
    | `pending`   | 注文受付 | 注文作成時         | `confirmed`        |
    | `confirmed` | 確認済み | 支払い完了時       | `shipped`          |
    | `shipped`   | 発送済み | 発送処理完了時     | `delivered`        |
    | `delivered` | 配送完了 | 配送完了確認時     | -                  |

## エラー・例外の定義

**対象**: システムで定義されているエラークラス

**定義すべき項目**:

- クラス名
- 発生条件
- 対処方法（ユーザー向け/開発者向け）
- ログレベル

## 用語の保守・更新

### 用語追加のタイミング

**追加すべき時**:

- 新しい概念が導入された
- チームメンバーから質問があった用語
- ドキュメントで3回以上出現する用語
- 外部サービスやAPIを統合した時

**追加不要な時**:

- プログラミング一般の用語（変数、関数など）
- 一度しか使わない一時的な用語
