# パフォーマンス

処理効率・リソース使用量・応答速度に関する問題を検証します。

## データベースパフォーマンス

| パターン               | 確認ポイント                                                             |
| ---------------------- | ------------------------------------------------------------------------ |
| **N+1クエリ**          | ループ内でDBクエリを個別発行していないか。JOINやバッチクエリで置き換える |
| **インデックス不足**   | WHERE句・ORDER BY句で使用するカラムにインデックスがあるか                |
| **過剰取得**           | SELECT \*で全カラムを取得していないか。必要なカラムだけを指定する        |
| **コネクションプール** | リクエストごとにDB接続を作成していないか。プーリングを使用する           |

## アルゴリズムの計算量

以下のO(n²)パターンを検出した場合は指摘する。

- **ネストしたループ**: 同一コレクションへの二重ループ。Mapやセットで線形化できないか
- **ループ内のソート**: ソートをループの外に出せないか
- **繰り返しの線形探索**: `array.find()`をループ内で呼ぶ場合、事前にMapを構築すべき
- **複数回のフィルタリング**: 同じデータへの複数パスを1回に統合できないか

## メモリ管理

### メモリリークの原因パターン

- 登録したまま解除されないイベントリスナー
- クリアされないタイマー・インターバル
- 肥大化の制限がないキャッシュ（エビクションポリシーの欠如）
- クローズされないリソース（ファイルハンドル、DB接続等）

### ストリーム vs バッチ

- 大量データ（ファイル、DBレコード等）を一括でメモリに読み込んでいないか
- ストリーミングやチャンク処理で逐次処理できないか検討する

## フロントエンドパフォーマンス

| 観点                     | 確認ポイント                                                                   |
| ------------------------ | ------------------------------------------------------------------------------ |
| **バンドルサイズ**       | 未使用の依存をインポートしていないか。大きなライブラリに軽量な代替がないか     |
| **遅延読み込み**         | ルート・コンポーネントをオンデマンドで読み込んでいるか。コード分割しているか   |
| **不要な再レンダリング** | 毎回変わるprops、メモ化の欠如、親の状態変更による不要な子の再描画がないか      |
| **画像最適化**           | リサイズ・圧縮・WebP形式を使用しているか。Below-the-foldの画像は遅延読み込みか |

## キャッシュ戦略

### キャッシュすべき場面

- 高コストな計算（重いクエリ、データ変換）
- 頻繁にアクセスされ、変更頻度が低いデータ
- 外部APIの呼び出し結果

### 無効化パターン

| パターン             | 用途                                  |
| -------------------- | ------------------------------------- |
| **時間ベース (TTL)** | 厳密さが不要なデータ向け              |
| **イベントベース**   | CUD操作時に対応するキャッシュを無効化 |
| **依存ベース**       | 関連するキャッシュをまとめてクリア    |

### キャッシュスタンピード

キャッシュ失効時に大量リクエストがバックエンドに殺到する問題。ロック（1リクエストだけ取得、他は待機）や確率的早期再取得で防止する。

## チェック項目

- [ ] ループ内のDBクエリ・APIコールがないか（N+1問題）
- [ ] O(n²)以上の計算量を持つアルゴリズムがないか
- [ ] メモリリークの原因パターンに該当しないか
- [ ] 大量データの一括読み込みをしていないか
- [ ] フロントエンドのバンドルサイズ・再レンダリングが適切か
- [ ] キャッシュが必要な箇所に適切に導入されているか

## レビュー手順

1. 変更ファイルのdiffからデータ処理・DB操作・レンダリングに関する箇所を特定する
2. 上記の各カテゴリのパターンに照らして問題を検出する
3. 問題を検出した場合は具体的な改善手法を提案する

## 指摘の観点

- **N+1クエリ**: ループ内でDBクエリを個別発行している
- **非効率なアルゴリズム**: O(n²)パターンが検出された
- **メモリリーク**: リソースの解放漏れやリスナーの未解除
- **ブロッキング処理**: 非同期にすべき重い処理が同期的に実行されている
