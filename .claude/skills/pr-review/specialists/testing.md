# テスト

テストの網羅性・品質・信頼性を評価します。

## テストピラミッド

適切なテスト配分になっているかを確認する。

| レベル                       | 割合目安 | 特徴                                                   |
| ---------------------------- | -------- | ------------------------------------------------------ |
| **ユニットテスト**           | 70%      | 高速・独立・個々の関数やメソッドを検証。テストの土台   |
| **インテグレーションテスト** | 20%      | コンポーネント間の連携（DB、API、サービス間）を検証    |
| **E2Eテスト**                | 10%      | 重要なユーザーフローのみ。保守コストが高いため最小限に |

## FIRST原則

テストが以下の原則を満たしているかを確認する。

- **Fast（高速）**: テストは秒単位で完了するか。遅いテストはスキップされがち
- **Independent（独立）**: 各テストが実行順序に依存せず単独で動くか。共有状態がないか
- **Repeatable（反復可能）**: 同じ環境で何度実行しても同じ結果が出るか
- **Self-Validating（自己検証）**: pass/failが自動判定されるか。手動でログを確認する必要がないか
- **Timely（適時）**: テストがコードと同時に書かれているか。後回しにされていないか

## テストの構造: Arrange-Act-Assert

各テストが以下の3フェーズで明確に構成されているかを確認する。

- **Arrange（準備）**: テストデータ・モックのセットアップ。必要最小限に保つ
- **Act（実行）**: テスト対象の操作を1つだけ実行する
- **Assert（検証）**: 期待値を具体的に検証する。「nullでない」だけでなく実際の値を検証

## テストダブルの適切な使い分け

| 種類      | 用途                       | 使うべき場面                           |
| --------- | -------------------------- | -------------------------------------- |
| **Dummy** | 未使用パラメータの穴埋め   | 引数が必要だが値は関係ないとき         |
| **Stub**  | 固定の戻り値を返す         | 依存先の出力を制御したいとき           |
| **Fake**  | 簡易的な動作する実装       | インメモリDBなど、本物が不要なとき     |
| **Mock**  | 呼び出しの検証             | 依存先が正しく呼ばれたか確認したいとき |
| **Spy**   | 呼び出しを記録して後で検証 | 振る舞いと結果の両方を追跡したいとき   |

原則: **Stubは状態を検証、Mockは振る舞いを検証**する。過剰なモックは実際の問題を隠す。外部依存のみモックし、内部ロジックのモックは避ける。

## コンポーネントテスト

フロントエンドのコンポーネントに対するテストの確認観点。

- **ユーザー視点のテスト**: 内部state/propsではなく、描画結果やユーザー操作（クリック、入力）をテストしているか
- **Testing Libraryの原則**: `getByRole`, `getByText`等のアクセシビリティベースのクエリを使用しているか。`getByTestId`は最終手段
- **インタラクションテスト**: ユーザーイベント（`userEvent.click`等）を使い、実際の操作をシミュレートしているか
- **スナップショットテストの乱用**: 大きなスナップショットは変更のたびに壊れる。意図的な差分の確認が困難。小さく限定的に使う

## テストのアンチパターン

以下のパターンを検出した場合は指摘する。

- **実装詳細のテスト**: 内部構造に依存したテストはリファクタで壊れる。観察可能な振る舞いをテストすべき
- **フレイキーテスト**: タイミング依存・外部システム依存で結果が不安定。根本原因の解決が必要
- **テスト間の依存**: 共有セットアップや実行順序への依存。各テストが自己完結すべき
- **過剰なモック**: すべてをモックすると実際の不具合を見逃す。外部依存のみモックする
- **アサーションの欠如**: テストが実行されるが何も検証していない。偽の安心感を生む

## チェック項目

- [ ] 変更に対応するテストが追加・更新されているか
- [ ] テストピラミッドの配分が適切か（ユニット > インテグレーション > E2E）
- [ ] FIRST原則を満たしているか（特にIndependent・Repeatable）
- [ ] Arrange-Act-Assertの構造が明確か
- [ ] エッジケース（境界値、空値、大量データ等）が考慮されているか
- [ ] 上記のアンチパターンに該当しないか

## レビュー手順

1. 変更ファイルに対応するテストファイルの有無を確認する
2. 新規・変更されたロジックに対するテストケースを評価する
3. FIRST原則・AAAパターン・テストダブルの使い方を確認する
4. カバーされていないパターンやアンチパターンを特定する

## 指摘の観点

- **テストの欠落**: 新機能・修正に対するテストが存在しない
- **エッジケース未考慮**: 境界値や異常系のテストが不足している
- **脆いテスト**: 実装の内部詳細に依存しておりリファクタで壊れやすい
- **不明確なテスト名**: テストが何を検証しているか名前から判断できない
- **過剰なモック**: 外部依存以外をモックしており実際の問題を隠している
