# 詳細設計書作成スキル

高品質な詳細設計書を作成するためのスキルです。基本設計書で定義されたアーキテクチャに基づき、各機能の実装レベルの設計を詳細化します。

## 前提条件

- **必須**: `docs/requirements.md`（要件定義書）が作成されている
- **必須**: `docs/basic-design.md`（基本設計書）が作成されている

## 出力ルール

`../../rules/document-generation-guide.md` を参照。

**対象ファイル**: `docs/detailed-design.md`

## 参照ファイル

- **テンプレート**: `./assets/template.md`
- **作成ガイド**: `./references/guide.md`

## 詳細設計書作成プロセス

### 1. 入力の確認

- `docs/requirements.md` を読み、機能要件・非機能要件を把握する
- `docs/basic-design.md` を読み、アーキテクチャ・データモデル・インターフェース方針を把握する
- 既存の `docs/detailed-design.md` があるか確認。ある場合は `../../rules/document-generation-guide.md` の「既存ドキュメント最優先」ルールに従い、既存の節構造・表記・言語を維持したまま更新する
- 前提ドキュメントが存在しない場合は下記トラブルシューティングに従う（skill を中止）

### 2. 詳細設計書ドラフトの生成

`docs/requirements.md` と `docs/basic-design.md` の内容をもとに、テンプレート（`./assets/template.md`）に従って詳細設計書を生成します。詳細な記述ポイントは `./references/guide.md` を参照してください。

行数の目安は **300-400 行以内（上限目安、簡潔優先）**。超過する場合は `../../rules/document-generation-guide.md` に従いサブディレクトリ（`docs/design/`）に分割する。

### 3. 詳細設計書のレビューと改善

生成された詳細設計書を以下 5 観点でレビューし、問題があれば修正する:

1. API / インターフェース設計の妥当性（エンドポイント・DTO・バリデーションが具体的か）
2. クラス・モジュール設計（責務分離、依存方向が明確か）
3. DB スキーマ整合（基本設計のデータモデルと齟齬がないか、インデックス戦略が妥当か）
4. エラーハンドリング・例外方針（業務例外・システム例外の区分、リトライ方針）
5. 要件トレーサビリティ（基本設計の各コンポーネントが詳細化されているか）

**終了条件**: 品質チェックリスト（本 skill 末尾）の全項目が ○ になり、かつユーザー（人間）が承認した時点で終了する。

**レビュー結果の扱い**: レビュー結果・改善ログは `docs/detailed-design.md` 本体には含めず、会話上のログとしてユーザーに提示する。

**注意点**: AI のレビューを鵜呑みにせず、最終的な判断は必ず人間が行う

## 既存ドキュメント更新時の整合判断

既存の `docs/detailed-design.md` があり、新しい基本設計と関係する場合:

- **基本設計に既存詳細の廃止が明示されている**: 置換（既存記述を削除し新詳細で差し替え）
- **廃止記述がない**: 併存（既存を残し、新詳細を追記して関係を注記）
- **スタイル整合**: 既存が英語節・箇条書き主体なら新規追記もそれに合わせる（template の日本語節構造を強制しない）

## 品質チェックリスト

### API / インターフェース設計

- [ ] エンドポイント一覧とパス・HTTP メソッドが明記されているか
- [ ] リクエスト/レスポンス DTO の型が定義されているか
- [ ] バリデーションルール（必須・上限・フォーマット）が具体的か

### クラス・モジュール設計

- [ ] 各モジュール/クラスの責務が 1 行で説明できるレベルで明記されているか
- [ ] 依存方向が基本設計のレイヤー構造と整合しているか

### DB スキーマ

- [ ] テーブル定義（カラム・型・制約）が明記されているか
- [ ] 主要クエリのインデックス戦略が記載されているか
- [ ] マイグレーション方針が定義されているか

### エラーハンドリング

- [ ] 業務例外とシステム例外の区分が明確か
- [ ] リトライ・タイムアウト・サーキットブレーカー等のポリシーが定義されているか
- [ ] エラーレスポンス形式が統一されているか

### トレーサビリティ

- [ ] 基本設計書の各コンポーネントがこのドキュメントで詳細化されているか

## トラブルシューティング

- **前提ドキュメントが不足**: `docs/requirements.md` と `docs/basic-design.md` が必要です。`requirements-definition` / `basic-design` skill で作成してから再実行してください。
- **既存の設計書との競合**: `../../rules/document-generation-guide.md` の優先順位ルールに従ってください（既存ドキュメント最優先）。
