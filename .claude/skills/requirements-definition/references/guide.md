# 要件定義書作成ガイド

要件定義書作成時の詳細なポイントとセクション別の記述ガイドです。

## 要件定義書作成の重要なポイント

### 1. 具体性と測定可能性

すべての要件は具体的で測定可能でなければなりません。

**悪い例**:

- システムは高速である必要がある
- ユーザーは使いやすいと感じる

**良い例**:

- API応答時間: 200ms以内（P95）
- 新規ユーザーが5分以内に基本操作を習得できる（ユーザビリティテストで測定）

### 2. ユーザー中心設計

すべての機能は、明確なユーザーの課題を解決するものでなければなりません。

**ユーザーストーリーのフォーマット**:

```
[ユーザー]として、[目的]のために、[機能]が欲しい
```

**例**:

```
管理者として、チームの進捗を把握するために、
ダッシュボードでプロジェクト全体の状況を確認したい
```

### 3. 優先順位の明確化

すべての機能に優先度を設定します:

- **P0（必須）**: MVPに含める機能。これがないとプロダクトとして成立しない
- **P1（重要）**: 初期リリース後すぐに追加すべき機能
- **P2（できれば）**: 将来的に追加を検討する機能

## 主要セクション詳細

### 1. プロダクト概要

#### 構成要素

1. **名称**: プロダクト名とサブタイトル
2. **プロダクトコンセプト**: 3つの主要コンセプト
3. **プロダクトビジョン**: 目指す世界観を3-5文で
4. **目的**: 具体的な目的のリスト

#### 例

    ### 名称

    **ProjectHub** - チーム向けプロジェクト管理プラットフォーム

    ### プロダクトコンセプト

    - リアルタイム共同編集: チームメンバーが同時にドキュメントやタスクを編集
    - 自動進捗トラッキング: タスクの状態変更から進捗を自動算出
    - シンプルなインターフェース: 必要最低限の操作で日常業務を完結

**具体的な価値提案を含める**

悪い例:

```
便利なプロジェクト管理ツールを作る
```

良い例:

```
チームの進捗管理とコミュニケーションを一元化するプラットフォーム。

提供する価値:
- 情報の分散を防止（チャット・タスク・ドキュメントを統合）
- 進捗報告の自動化（手動レポート作成の工数を80%削減）
- オンボーディングの高速化（新メンバーが1日で業務開始可能）
```

### 2. ターゲットユーザー（ペルソナ）

#### 必須要素

1. **基本属性**: 年齢、職業、経験年数
2. **技術スタック/利用環境**: 使用しているツールや環境
3. **現在の課題**: 具体的な痛み点
4. **期待する解決策**: どうなりたいか
5. **典型的なワークフロー**

#### 例

    ### プライマリーペルソナ: 鈴木花子（34歳、プロジェクトマネージャー）

    - 5-10人のチームを複数プロジェクトで管理
    - Slack、Google Workspace、Jiraを併用
    - ツール間の情報分散で進捗把握に時間がかかる
    - 週次レポートの手動作成に毎週2時間を費やしている

    ### セカンダリーペルソナ: 山田次郎（26歳、ジュニアエンジニア）

    - チーム内でタスクを割り当てられる側
    - 自分の担当タスクと期限を素早く確認したい
    - プライマリーとの違い: 管理ではなく実行が主な関心事

セカンダリーペルソナはプライマリーより簡潔に記述し、プライマリーとの違いを明示してください。

### 3. 成功指標（KPI）

#### SMART原則

- **S**pecific（具体的）: 何を測定するか明確
- **M**easurable（測定可能）: 数値で測定できる
- **A**chievable（達成可能）: 現実的な目標
- **R**elevant（関連性）: ビジネス目標と関連
- **T**ime-bound（期限）: 達成期限を設定

#### 例

    - 週次アクティブチーム数: 50チーム（6ヶ月後）
    - タスク完了率: 75%以上
    - 平均オンボーディング時間: 1日以内

### 4. 機能要件

#### コア機能（MVP）

各機能には以下を含めます:

- ユーザーストーリー
- 受け入れ条件（チェックリスト形式）
- 優先度（P0/P1/P2）

**フォーマット**:

    ### [機能名]

    ユーザーストーリー:
    [ユーザー]として、[目的]のために、[機能]が欲しい

    受け入れ条件:

    - [ ] 条件1（測定可能）
    - [ ] 条件2（測定可能）

    優先度: P0（必須） / P1（重要） / P2（できれば）

### 5. 非機能要件

測定可能な形で記述します:

**例**:

    ### パフォーマンス

    - ページ読み込み時間: 2秒以内（3G回線を除く）
    - API応答時間: 200ms以内（P95）

    ### ユーザビリティ

    - 新規ユーザーが5分以内に基本操作を習得できる
    - モバイル端末でも主要機能が利用可能

    ### 信頼性

    - 稼働率: 99.9%以上
    - データ損失ゼロ（自動バックアップ）

    ### セキュリティ

    - 認証情報の暗号化保存
    - 操作ログの監査証跡
