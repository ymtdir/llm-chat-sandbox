# 要件定義書作成スキル

高品質な要件定義書を作成するためのスキルです。

## 前提条件

- **推奨**: `docs/ideas/*.md` にアイデアや要件メモが配置されている
- ファイルがない場合は対話形式で要件をヒアリングする

## 出力ルール

`../../rules/document-generation-guide.md` を参照。

**対象ファイル**: `docs/requirements.md`

## 参照ファイル

- **テンプレート**: `./assets/template.md`
- **作成ガイド**: `./references/guide.md`

## 要件定義書作成プロセス

### 1. 入力の確認

- `docs/ideas/` 内のマークダウンファイルを全て読む
- 既存の `docs/requirements.md` があるか確認。ある場合は `../../rules/document-generation-guide.md` の「既存ドキュメント最優先」ルールに従い、既存の節構造・表記・言語を維持したまま更新する
- `docs/ideas/` が空または未存在の場合は下記「対話形式ヒアリング」節に進む

### 2. 要件定義書ドラフトの生成

`docs/ideas/*.md` の内容をもとに、テンプレート（`./assets/template.md`）に従って要件定義書を生成します。詳細な記述ポイントは `./references/guide.md` を参照してください。

行数の目安は **300-400 行以内（上限目安、簡潔優先）**。下限は要求しない。超過する場合は `../../rules/document-generation-guide.md` に従いサブディレクトリに分割する。

### 3. 要件定義書のレビューと改善

生成された要件定義書を以下 5 観点でレビューし、問題があれば修正する:

1. プロダクトビジョンは明確か
2. ターゲットユーザーは具体的か
3. 成功指標は測定可能か
4. 機能要件は実装可能なレベルまで詳細化されているか
5. 非機能要件は網羅されているか

**終了条件**: 品質チェックリスト（本 skill 末尾）の全項目が ○ になり、かつユーザー（人間）が承認した時点で終了する。自己レビューのみで無限ループしない。

**レビュー結果の扱い**: レビュー結果・改善ログは `docs/requirements.md` 本体には含めず、会話上のログとしてユーザーに提示する（本体は成果物として簡潔に保つ）。

**注意点**: AIのレビューを鵜呑みにせず、最終的な判断は必ず人間が行う

## 対話形式ヒアリング

`docs/ideas/` が空または未存在の場合、以下の順でユーザーにヒアリングする。

### 事前分岐

- **構想がある程度まとまっている**（ユーザーが口頭で話せる状態）: 即ヒアリングに入る
- **完全に白紙**: まず `docs/ideas/` にラフなメモを書き出すよう推奨してから再開

### ヒアリング順序

`./assets/template.md` の節順（プロダクト概要 → ターゲットユーザー → 成功指標 → 機能要件 → 非機能要件 → スコープ外）で進める。節ごとに区切り、前節が埋まってから次節へ進む。

### 曖昧回答への対処

ユーザー回答が `./references/guide.md` の「良い例」水準（具体・測定可能・SMART 準拠）を満たさない場合、1 回だけ掘り下げ質問を返す。2 回目でも曖昧なら推測で補完しドラフトに反映、レビュー段階でユーザー確認を仰ぐ。

### 任意節の扱い

セカンダリーペルソナ・スコープ外・非機能要件の各カテゴリは「該当する場合」が任意。ユーザーに「必要か」を聞き、不要ならその節は空欄または省略する（template の節は残しつつ「該当なし」と明記）。

## 既存ドキュメント更新時の整合判断

既存の `docs/requirements.md` があり、新しい ideas が既存機能と関係する場合:

- **idea に既存機能の廃止が明示されている**: 置換（既存記述を削除し新機能で差し替え）
- **idea に廃止記述がない**: 併存（既存を残し、新機能を追記して関係を注記）
- **優先度の整合**: 既存が優先度未記載の場合、新規のみ優先度を付与する（全体を書き換えない）

## 品質チェックリスト

### ビジョンと目標

- [ ] プロダクトビジョンが明確で測定可能か
- [ ] 提供する具体的な価値が定義されているか
- [ ] ターゲット市場が明確か

### ターゲットユーザー

- [ ] ペルソナが具体的に定義されているか
- [ ] 現在の課題と期待する解決策が明確か

### 成功指標

- [ ] SMART原則に従ったKPIが定義されているか
- [ ] 測定方法が明確か

### 機能要件

- [ ] すべての機能がユーザーストーリー形式で記述されているか
- [ ] 受け入れ条件が測定可能な形で定義されているか
- [ ] 優先度(P0/P1/P2)が明確に設定されているか

### 非機能要件

- [ ] パフォーマンス基準が具体的な数値で定義されているか
- [ ] 信頼性とセキュリティ要件が明確か

## トラブルシューティング

### docs/ideas/ にファイルがない

ユーザーに `docs/ideas/` へアイデアや要件メモの配置を依頼してください。ファイルがない場合は対話形式で要件をヒアリングします。

### 要件定義書の粒度が不適切

- 粗すぎる場合: `./references/guide.md` のセクション別ガイドを参照して詳細化
- 細かすぎる場合: 詳細設計書（detailed-design）に委譲すべき内容を分離
