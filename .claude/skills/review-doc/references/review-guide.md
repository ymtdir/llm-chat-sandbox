# ドキュメントレビューガイド

## 共通チェック項目

あらゆるドキュメントに適用します。

- [ ] ドキュメントの目的が冒頭で明確か
- [ ] 対象読者にとって必要な情報が揃っているか
- [ ] 曖昧な表現がないか（「高速」「適切」等の未定量な表現）
- [ ] 構造が論理的で読みやすいか
- [ ] 用語が一貫しているか
- [ ] 誤字脱字がないか

## プロジェクトドキュメント固有の観点

対応するスキルのテンプレート・ガイドがある場合は、それも参照してレビューしてください。

### 要件定義書

- [ ] ターゲットユーザーが具体的か
- [ ] 成功指標が測定可能か
- [ ] 機能要件に受け入れ条件があるか
- [ ] 優先度が設定されているか
- [ ] スコープ外が明示されているか

### 基本設計書

- [ ] 技術選定に理由があるか
- [ ] アーキテクチャの選定理由が明確か
- [ ] データモデルがあるか
- [ ] セキュリティが考慮されているか

### 詳細設計書

- [ ] データモデルが型定義で記述されているか
- [ ] コンポーネントのインターフェースが定義されているか
- [ ] ユースケースのシーケンス図があるか
- [ ] エラーハンドリングが網羅されているか

### 開発ガイドライン

- [ ] リポジトリ構造が視覚化されているか
- [ ] 命名規則が明確か
- [ ] ブランチ戦略・コミット規約が決まっているか
- [ ] テスト戦略が設定されているか

## レビューの姿勢

- **建設的**: 批判ではなく改善提案を行う
- **具体的**: 「どこが」「なぜ」「どう改善するか」を明示
- **バランス**: 良い点も必ず指摘する
